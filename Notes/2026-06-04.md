# Node.js Backend Notes - 2026-06-04

Today I moved the journal backend from local JSON files to MongoDB using Mongoose.

## What Changed

Earlier, the model files used local JSON files as a simple database:

```txt
users.json
data.json
```

The old flow was:

```txt
read JSON file
parse it into an array
find/update/delete data inside the array
write the whole array back to the file
```

Now the backend talks to MongoDB through Mongoose:

```txt
request
  -> route
  -> controller
  -> service
  -> Mongoose model
  -> MongoDB
```

The controller and service layers stayed mostly the same. The main change happened inside the model layer.

## MongoDB Connection

The backend reads the MongoDB URL from `.env`:

```txt
MONGODB_URI=...
```

Then the app connects to MongoDB before starting the Express server.

This means the database URL is not hardcoded in the codebase.

## Schema And Model

I learned that Mongoose has two important ideas:

```txt
Schema = the shape and rules of the data
Model = the object used to create, find, update, and delete data
```

Example:

```js
const User = mongoose.model('User', userSchema)
```

After creating a model, we can use methods like:

```js
User.findOne(...)
User.create(...)
User.findByIdAndDelete(...)
```

## Replacing JSON Functions

Old JSON logic looked like this:

```js
const users = await readAll()
return users.find(user => user.username === username)
```

MongoDB logic now looks like this:

```js
const user = await User.findOne({ username })
```

This means:

```txt
MongoDB, find one user where username matches this value.
```

For journals, this query gets all journals owned by one user:

```js
Journal.find({ userId })
```

## MongoDB IDs

The old JSON database used numeric IDs created with:

```js
Date.now()
```

MongoDB creates its own `_id` automatically.

Because MongoDB IDs are strings/ObjectIds, I removed code like:

```js
Number(req.params.id)
```

Now IDs are passed around as strings.

## Returning Clean Data To Frontend

MongoDB returns `_id`, but the frontend expects `id`.

So the model converts Mongo documents into frontend-friendly objects:

```js
return {
  id: journal._id.toString(),
  userId: journal.userId,
  title: journal.title,
  date: journal.date,
}
```

This keeps MongoDB details inside the backend.

## Validation

Mongoose validates data using the schema.

For create operations, validation runs automatically:

```js
Journal.create(...)
```

For update operations, we need:

```js
runValidators: true
```

Example:

```js
Journal.findByIdAndUpdate(id, allowedChanges, {
  new: true,
  runValidators: true,
})
```

I also learned to allow only safe fields during update:

```js
const allowedChanges = {}
if (changes.title !== undefined) allowedChanges.title = changes.title
```

This prevents users from editing protected fields like `userId` or `date`.

## User And Journal Relationship

One user can have many journals.

That relationship is stored like this:

```txt
User
  -> many Journals
```

Each journal stores the owner ID:

```js
userId
```

So this query finds only one user's journals:

```js
Journal.find({ userId })
```

This is called referencing.

## Indexes

I learned that indexes help MongoDB find data faster.

The journal app often does this:

```js
Journal.find({ userId }).sort({ createdAt: -1 })
```

So a useful index would be:

```js
journalSchema.index({ userId: 1, createdAt: -1 })
```

This matches the query shape:

```txt
find by userId
sort newest first
```

Indexes speed up reads, but they also add a small cost during writes, so I should add them only when they match real queries.

## Important Security Note

Right now the app still trusts `userId` from the frontend.

For real security, the next step is authentication with JWT:

```txt
login
  -> backend returns token
  -> frontend sends token with requests
  -> backend verifies token
  -> backend knows the real userId
```

Then journal updates can check:

```js
{ _id: journalId, userId: loggedInUserId }
```

That means:

```txt
Only update this journal if it belongs to the logged-in user.
```

## Main Takeaway

Today's biggest lesson:

```txt
The app structure did not need to change much.
Only the storage layer changed from JSON files to MongoDB documents.
```

The frontend still calls the same backend APIs. Controllers and services still do their jobs. The model layer now talks to MongoDB instead of reading and writing local files.
